1 · Was ist SSH überhaupt?

SSH (Secure Shell) ist die Art, wie sich ein Computer bei einem anderen sicher anmeldet, um dort Befehle auszuführen oder Dateien zu lesen. Wenn Michael „auf den Server geht", benutzt er SSH.

Statt eines Passworts nutzt man ein Schlüsselpaar — das wichtigste Bild:

TeilWas es istWo es liegtBild
Privater SchlüsselDie geheime HälfteBeim Anmelder (Michaels Laptop / bei uns: GitHub)Dein Hausschlüssel — gibst du niemals her
Öffentlicher SchlüsselDie offene HälfteAuf dem Server in der Liste authorized_keysDas Türschloss — darf jeder sehen

Bei der Anmeldung stellt der Server eine Rechenaufgabe, die nur mit dem privaten Schlüssel lösbar ist. Der private Schlüssel verlässt dabei nie den Anmelder — es wird kein Passwort über die Leitung geschickt.

Merksatz: Öffentlicher Schlüssel = Schloss (darf rumliegen). Privater Schlüssel = Schlüssel (streng geheim). Wer den privaten Schlüssel hat, kommt rein.

2 · Was wir gerade machen (Ist-Zustand)

2a · Der CISA-KEV-Scanner (neu, seit heute)

Damit unser Sicherheits-Scanner in GitHub Actions die echten Software-Versionen lesen kann (WordPress, Plugins, Gambio …), bekam er einen SSH-Zugang. Der private Schlüssel liegt als GitHub-Secret PROFIHOST_SSH_KEY. Er liest nur ein paar Versions-Dateien (cat, grep, php -v).

2b · Michaels Schlüssel (gewachsen)

Michael nutzt SSH für viele Systeme — Schnittstellen, Wartung, Deployments, Backups. In der Praxis ist oft ein einziger, langlebiger Schlüssel im Einsatz, der überall passt. Bequem, aber der schwächste Punkt.

Das versteckte Problem: „Nur-lesend" ist aktuell nur eine Vereinbarung in unserem Code — keine Regel, die der Server erzwingt.

Der Schlüssel selbst hat sehr wahrscheinlich vollen Shell-Zugang. Würde der GitHub-Secret jemals geleakt, könnte ein Angreifer damit nicht nur lesen, sondern alles tun, was der Schlüssel darf. Echte Sicherheit kommt erst, wenn der Server den Schlüssel einschränkt (Teil 6).

3 · Was bedeutet „SSH rotieren"?

Rotation = den Schlüssel regelmäßig austauschen, so wie man ein Passwort regelmäßig ändert. Genau deine Frage.

1 · Neues Schlüsselpaar erzeugen (neues Schloss + neuer Schlüssel)
2 · Neues Schloss zur authorized_keys hinzufügen
3 · Neuen privaten Schlüssel überall eintragen (z. B. GitHub-Secret)
4 · Testen, dass alles läuft
5 · Altes Schloss entfernen — alter Schlüssel ist ab jetzt tot
Ohne RotationMit Rotation
Ein geleakter Schlüssel funktioniert für immer — bis jemand zufällig merkt, dass er weg ist. Ein geleakter Schlüssel funktioniert nur bis zur nächsten Rotation — danach wertlos.

Analogie: Ein Hotel wechselt nach jedem Gast die Türcodes. Selbst wenn ein Gast den Code mitnimmt, nützt er ihm beim nächsten Mal nichts mehr.

Rotation verkleinert das Zeitfenster, in dem ein gestohlener Schlüssel Schaden anrichten kann — sie verhindert den Diebstahl nicht, sorgt aber dafür, dass er nicht ewig nachwirkt.

4 · Die Techniken im Vergleich

Von „bequem aber unsicher" nach „sicher aber aufwändiger":

A · Ein Schlüssel für alles (heutiger De-facto-Stand)

Ein langlebiger „Generalschlüssel". Leakt er, ist alles offen, und unbegrenzt. Man weiß nie, wer was getan hat. ❌ Veraltet.

B · Ein eigener Schlüssel pro Zweck

Ein Schlüssel nur für den Scanner, einer für Schnittstellen, einer für Backups. Leakt einer, ist nur dieser eine Bereich betroffen. ✅ Prinzip „Least Privilege".

C · Eingeschränkter Schlüssel (forced command)

In der authorized_keys schreibt man vor den Schlüssel, was er darf:

command="/usr/local/bin/inventar-readonly.sh",restrict ssh-ed25519 AAAA... scanner@github

Jetzt ist „nur-lesend" vom Server erzwungen. Selbst mit gestohlenem Schlüssel kann ein Angreifer nur das eine Skript auslösen — keine Shell, kein Löschen. ✅✅ Größter Gewinn für unseren Scanner-Fall.

D · Rotation (kombinierbar mit B und C)

Schlüssel regelmäßig (z. B. quartalsweise) oder bei Verdacht tauschen. Begrenzt die Lebensdauer eines Leaks. ✅ Standard-Hygiene.

E · SSH-Zertifikate (CA-signiert, kurzlebig)

Eine zentrale „Ausweisstelle" signiert kurzlebige Ausweise (z. B. 8 Std). Kein Aufräumen von authorized_keys, Rotation passiert automatisch durch Ablauf. Goldstandard großer Teams — für ein 2-Personen-Team aber vermutlich Overkill. ✅✅✅ State of the Art.

5 · Bezug zu unserem neuen Scanner

Der CISA-KEV-Scanner braucht vom Server nur sehr wenig: ein paar Versions-Dateien lesen. Perfekter Kandidat für B + C:

  • B: ein eigener Schlüssel scanner@github, getrennt von Michaels Generalschlüssel.
  • C: dieser Schlüssel darf auf dem Server nur ein Lese-Skript ausführen.

Dann gilt: Selbst wenn der GitHub-Secret jemals leakt, kann ein Angreifer nur die Versionsnummern lesen — also genau das, was eh in der täglichen Mail steht. Kein Schaden möglich.

6 · Empfehlung & nächster Schritt

ICE = Impact × Confidence × Ease / 100. MUST ≥ 5,0 · SHOULD ≥ 3,0.

MaßnahmeICEICEStufe
1 · Eigener, eingeschränkter Lese-Schlüssel für den Scanner (B+C)9975,7MUST
2 · Schlüssel pro Zweck trennen (Generalschlüssel auflösen)8853,2SHOULD

Konkreter nächster Schritt (für Michael, ~15 Min)

# 1) Neues Schlüsselpaar
ssh-keygen -t ed25519 -C "scanner@github" -f scanner_key

# 2) Auf dem Server: Lese-Skript /usr/local/bin/inventar-readonly.sh anlegen
#    (liest wp-includes/version.php, Plugin-Header, Gambio release_info.php ...)

# 3) Öffentlichen Schlüssel in ~/.ssh/authorized_keys — MIT Einschränkung:
command="/usr/local/bin/inventar-readonly.sh",restrict ssh-ed25519 AAAA...scanner@github

# 4) Privaten Schlüssel als GitHub-Secret PROFIHOST_SSH_KEY hinterlegen
# 5) Workflow manuell starten und prüfen, dass Versionen kommen

Damit ist der Scanner-Zugang sicher gekapselt, und ihr habt Technik B + C produktiv im Einsatz — die Grundlage, auf der Rotation (D) später trivial wird.
